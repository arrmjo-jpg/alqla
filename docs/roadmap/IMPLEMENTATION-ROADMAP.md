## ملاحظات منهجيّة

1. **بند تضخّم ترميز Unicode لن يُنفَّذ مباشرة** — تحقّق (1.1) ثمّ تنفيذ
   (1.2) فقط بناءً على النتيجة.
2. **أي رقم أداء يجب تمييز Cold-cache عن Warm-cache صراحة.**
3. **لا قياس بلا خطّ أساس** — Sprint 0 قبل أي تنفيذ.
4. **الأولويّة: أخطاء حيّة مؤكَّدة ← SEO عالي الأثر ← شبكة أمان العقد.**
5. **لا Sprint "منتهٍ" لمجرّد انتهاء مهامّه** — Definition of Done صريح
   لكل Sprint.
6. **لا قرار "ضائع بين الاجتماعات"** — أي بند يعتمد على قرار بديل حقيقيّ
   (لا قرار هندسيّ روتينيّ) مسجَّل في
   [DECISION-LOG.md](../architecture/DECISION-LOG.md)، لا داخل هذه
   الوثيقة فقط.
7. **الرجوع (Rollback) يُجيب "ماذا لو فشل؟"؛ المراقبة (Monitoring) تُجيب
   "كيف سنعرف أنّه بدأ يفشل؟"** — حقلان مختلفان، ليسا مترادفَين.

## قائمة تحقّق موحَّدة لما بعد النشر (24 ساعة) — تنطبق على كل بند مصنَّف "Requires Production Monitoring"

- ✓ لا زيادة في معدَّل استجابات 5xx مقارنةً بمتوسّط الـ 24 ساعة السابقة
  للنشر.
- ✓ لا زيادة في أخطاء JS الحيّة (console errors) على الصفحات المتأثِّرة.
- ✓ لا زيادة في زمن استجابة الـ API (p50/p95) للـ endpoints المتأثِّرة.
- ✓ لم يُفعَّل أي Rollback (يدويّ أو تلقائيّ) خلال النافذة.

يُشار إليها أدناه اختصاراً بـ "**✅ نجاح 24 ساعة: القائمة الموحَّدة**"،
مع أي إضافة خاصّة بالبند إن وُجدت.
