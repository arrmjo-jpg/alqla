  و`/bourse` أولويّة أوضح لاحقاً.
- **الملفّات المتأثّرة**: يُضاف كقسم في `BASELINE-METRICS.md` (0.3).
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: الجدول أعلاه مُعتمَد رسميّاً (commit)، ويُستخدَم

# Sprint 1 — أعلى عائد (ROI)، أقلّ مخاطرة، أخطاء حيّة + SEO أوّلاً

### 1.1 التحقّق من فرضية تضخّم الترميز قبل أي تنفيذ (Unicode Encoding)

- **النوع**: Investigation. **المسؤول**: Backend.
- **السبب**: تضخّم 1.78–2.1× مقاس في حجم JSON الخام؛ الأثر الشبكيّ يعتمد
  على ضغط الإنتاج، غير مؤكَّد بعد.
- **التبعيّات**: لا شيء.
- **خطوات التحقّق**: (1) تأكيد إعداد Cloudflare لضغط `/api/*`. (2) `curl
  -H "Accept-Encoding: br, gzip" -D - <domain>/api/v1/ar/homepage` وقراءة
  `Content-Encoding` والحجم الفعليّ بعد الضغط. (3) توثيق النتيجة.
- **الوقت المتوقَّع**: 30–60 دقيقة. **المخاطرة**: صفر.
- **التأثير**: يُحدِّد أولويّة 1.2 الفعليّة.
- Backend؟ لا (فحص) — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: إجابة موثَّقة على: هل Brotli/gzip مفعَّل على
  `/api/*`؟ ما الفرق الفعليّ بالبايت بعد الضغط؟

### 1.2 تنفيذ `JSON_UNESCAPED_UNICODE` (بعد 1.1 فقط)

- **النوع**: Optimization (Backend). **المسؤول**: Backend.
- **السبب**: نتيجة 1.1. **التبعيّات**: **يعتمد على 1.1** (لا يبدأ قبل
  اكتمالها).
- **الوقت المتوقَّع**: 15–30 دقيقة. **المخاطرة**: منخفضة جدّاً.
- **التأثير**: تقليص كل استجابة JSON عربيّة تقريباً بنسبة ~45–55%.
- **الملفّات المتأثّرة**: نقطة تصيير JSON المركزيّة في Laravel.
- Backend؟ **نعم** — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ نعم.
- **معايير القبول**: استجابة أي endpoint عامّ تُظهِر UTF-8 خاماً لا
  `\uXXXX`؛ حجم استجابة الرئيسية/المقال منخفض بمقدار مطابق لقياس 1.1
  (±5%)؛ صفر اختلاف في اختبارات الواجهة القائمة.
- **↩ خطّة التراجع**: إزالة خيار `JSON_UNESCAPED_UNICODE` من نقطة
  التصيير → إعادة نشر (Redeploy) → **لا حاجة لـ Migration** — تغيير
  ترميز محض بلا أثر على قاعدة البيانات أو الحالة المخزَّنة.

### 1.3 API Contract Snapshot (شبكة أمان قبل أي تعديل)

- **النوع**: Safety-net / Observability. **المسؤول**: Backend + Frontend
  (مشترك — الباك إند يوفِّر الاستجابة، الواجهة تحفظها/تقارنها).
- **السبب**: مشروع Headless يعتمد كليّاً على ثبات عقد الـ API؛ تعديلات
  Sprint 1 وما بعده قد تُغيِّر شكل استجابة بصمت.
- **التبعيّات**: لا شيء — يُفضَّل تنفيذه **قبل** أي بند آخر في هذا
  Sprint يمسّ استجابة API (1.2، 2.7).
- **الإجراء الموسَّع (بعد ملاحظة أنّ الـ Regression قد يكون بالترويسات
  لا الـ JSON فقط)**: لكل من الـ 7 endpoints (`/{locale}/homepage`،
  `/{locale}/categories`، `/{locale}/articles/{slug}`،
  `/{locale}/writers/{id}`، `/{locale}/videos`، `/{locale}/videos/{slug}`،
  `/{locale}/reels`، `/{kind}` و`/{kind}/{slug}` للبثّ) يُحفَظ:
  - جسم الاستجابة الكامل (JSON خام).
  - **Status Code**.
  - **كل ترويسات الاستجابة**: `Content-Type`، `Cache-Control`، `ETag`
    (إن وُجدت)، `Last-Modified` (إن وُجدت)، `Vary`، `X-RateLimit-*`.
  - عبر: `curl -s -D headers.txt -o body.json <url>` لكل endpoint، حفظ
    الملفّين معاً باسم واضح + طابع زمنيّ + معرِّف commit.
- **الوقت المتوقَّع**: 1.5–2 ساعة (7 endpoints × جسم+ترويسات).
- **المخاطرة**: صفر. **التأثير**: **مرتفع طويل الأمد** — regression في
  `Cache-Control` (مثلاً) لا يظهر في مقارنة JSON وحدها، ويكفي لكسر
  استراتيجيّة الكاش بصمت.
- **الملفّات المتأثّرة**: `docs/roadmap/baselines/api-contracts/`
  (7×2 ملفّات: body + headers، لكل واحد).
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: 14 ملفّاً (جسم+ترويسات لكل من 7 endpoints) محفوظة،
  كل واحد بتاريخ ومعرِّف commit.

- Backend؟ **نعم إن حُذِف** — Frontend؟ لا — Migration؟ لا — إعادة
  اختبار؟ نعم إن حُذِف.
- **معايير القبول**: قرار مكتوب (إبقاء/توثيق/حذف).
- **↩ خطّة التراجع (إن حُذِف)**: استرجاع الملفّات من git + إعادة تسجيل
  الـ routes — **لا Migration** (لا تعديل بيانات).

### 3.7 إصلاح الجلب المضاعَف في `en/author/[id]`

- **النوع**: Optimization. **المسؤول**: Frontend.
- **التبعيّات**: لا شيء. **الوقت المتوقَّع**: 30 دقيقة. **المخاطرة**: منخفضة.
- **الملفّات المتأثّرة**: `app/en/author/[id]/page.tsx`.

- **معايير القبول**: نداء واحد فقط لكل تحميل صفحة.
- **↩ خطّة التراجع**: `git revert`.

### 3.8 معالجة معرّفات الأقسام المُثبَّتة بالكود (Multi-Site/White-Label)

- **النوع**: Architecture Decision (استراتيجيّ). **المسؤول**: Product +
  Backend + Frontend (قرار على مستوى إدارة المنتج).
- **شرط البدء**: مشروط بخطّة فعليّة لـ Multi-Site/White-Label.
- **التبعيّات**: قرار استراتيجيّ خارجيّ عن نطاق هذه الخارطة.
- **الوقت المتوقَّع**: أيّام. **المخاطرة**: مرتفعة إن نُفِّذ دون خطّة.
- Backend؟ محتمل — Frontend؟ **نعم** — Migration؟ محتمل — إعادة اختبار؟
  شامل.
- **معايير القبول**: لا ينطبق قبل قرار المضيّ.

### 3.9 اعتماد قياس Cold-vs-Warm Cache كممارسة قياسيّة دائمة

- **النوع**: Process / Methodology. **المسؤول**: الفريق كاملاً.
- **التبعيّات**: لا شيء. **الوقت المتوقَّع**: سياسة مستمرّة، لا تنفيذ
  لمرّة واحدة.
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ ينطبق على
  كل بند أداء مستقبليّ.
- **معايير القبول**: لا ينطبق (سياسة).
