# خارطة طريق التنفيذ — واجهة Next.js العامة

**وثيقة تنفيذ واحدة تُجمِّع نتائج التقارير الثلاثة**
([FRONTEND-AUDIT.md](FRONTEND-AUDIT.md)،
[FRONTEND-ARCHITECTURE-VALIDATION.md](FRONTEND-ARCHITECTURE-VALIDATION.md)،
[FRONTEND-RUNTIME-AUDIT.md](FRONTEND-RUNTIME-AUDIT.md)) في خطّة تنفيذ
مُرتَّبة بـ 4 مراحل (Sprint 0–3) حسب العائد (ROI)، لا الخطورة فقط.
**النسخة الحاليّة تدمج مراجعة CTO ثانية** (إعادة ترتيب أولويّات، إضافة
Sprint 0، Feature Flags، Risk Matrix، حقول Type/Acceptance Criteria
صريحة لكل بند).

كل بند: **النوع**، السبب، الوقت المتوقَّع، المخاطرة، التأثير، الملفّات
المتأثّرة، Backend/Frontend/Migration/إعادة اختبار، **معايير القبول**
(متى تُعتبَر المهمّة منتهية فعليّاً)، و**🚩 Feature Flag مقترَح** حيث
ينطبق.

## ملاحظات منهجيّة (محدَّثة بعد مراجعة CTO)

1. **بند تضخّم ترميز Unicode لن يُنفَّذ مباشرة** — تحقّق أوّلاً (1.1) ثمّ
   تنفيذ (1.1-ب) فقط بناءً على نتيجة التحقّق.
2. **أي رقم أداء (زمن استجابة، حجم) يُستخدَم في هذه الوثيقة أو مستقبلاً
   يجب تمييز Cold-cache عن Warm-cache صراحة** — لا افتراض أنّ رقماً
   واحداً يمثِّل "الأداء الحقيقيّ".
3. **لا قياس بلا خط أساس (Baseline)** — لهذا أُضيفت Sprint 0 قبل أي
   تنفيذ؛ بدونها لن نعرف فعليّاً ما الذي تحسَّن بعد Sprint 1.
4. **الأولويّة الآن لِـ: (أ) الأخطاء الحيّة المؤكَّدة، (ب) SEO عالي الأثر،
   (ج) شبكة الأمان (Contract Snapshot)** — قبل التحسينات الأدائيّة
   البحتة وقبل تنظيف الكود الميت، حتى لو كانت هذه الأخيرة رخيصة التنفيذ.

# Sprint 0 — Baseline & Observability (جديدة، قبل أي تعديل)

**الهدف الوحيد لهذه المرحلة: توثيق الحالة الحاليّة رقميّاً، بلا أي تغيير
كود، لتكون نقطة مقارنة لاحقة.** بدونها، أي ادّعاء "تحسَّن الأداء" بعد
Sprint 1 غير قابل للإثبات.

### 0.1 قياس Lighthouse (Desktop + Mobile)

- **النوع**: Observability.
- **السبب**: لا يوجد قياس Lighthouse رسميّ حتى الآن في أيّ من التقارير
  الثلاثة السابقة (الأدوات المتاحة في تلك الجلسة لم تتضمّن Lighthouse
  مباشرة).
- **الإجراء**: `npx lighthouse http://<production-or-staging-url> --preset=desktop --output=json --output-path=baseline-desktop.json` ثمّ نفس الأمر بلا `--preset=desktop` للجوّال. **يجب أن يكون القياس على بناء
  إنتاجيّ (`next build` + `next start`)، لا Dev** — درس ثابت من التقرير
  الثالث.
- **الوقت المتوقَّع**: 30 دقيقة.
- **المخاطرة**: صفر — قياس فقط.
- **التأثير**: يُتيح مقارنة قبل/بعد فعليّة لأوّل مرّة.
- **الملفّات المتأثّرة**: لا شيء بالكود؛ حفظ `baseline-desktop.json`/
  `baseline-mobile.json` في `docs/roadmap/baselines/`.
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا (هذا هو
  خطّ الأساس نفسه).
- **معايير القبول**: ملفّا JSON محفوظان يحتويان درجات Performance/SEO/
  Accessibility/Best-Practices لكلا الوضعين.

### 0.2 Web Vitals الحقيقيّة (LCP، CLS، INP)

- **النوع**: Observability.
- **السبب**: تقرير التشغيل الفعليّ رصد مؤشِّراً على CLS محتمل (وميض
  Recharts في `/bourse`) لكن بلا رقم CLS فعليّ مقاس.
- **الإجراء**: نفس تقرير Lighthouse (0.1) يتضمّن هذه القيم فعليّاً — هذا
  البند تحديداً لاستخراجها وتوثيقها بشكل منفصل وواضح لكل صفحة من الصفحات
  التمثيليّة (الرئيسية، مقال، قسم، `/bourse`).
- **الوقت المتوقَّع**: 15 دقيقة (استخراج من نتائج 0.1، لا قياس جديد).
- **المخاطرة**: صفر.
- **التأثير**: رقم CLS فعليّ لـ `/bourse` تحديداً يُحدِّد ما إذا كان بند
  3.2 (إصلاح Recharts) يستحقّ ترقية أولويّة أم يبقى Sprint 3.
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: جدول بـ LCP/CLS/INP لكل صفحة من الأربع.

### 0.3 حجم JS وعدد Requests وحجم JSON — توحيد فيما هو موجود بالفعل

- **النوع**: Observability (تجميع، لا قياس جديد بالكامل).
- **السبب**: جزء كبير من هذه البيانات **موجود بالفعل** من
  FRONTEND-RUNTIME-AUDIT.md (جدول First Load JS الكامل من `next build`،
  حجم JSON الفعليّ للرئيسية/المقال). هذا البند يُنسِّقها في ملفّ Baseline
  واحد رسميّ بدل بقائها موزَّعة داخل تقرير سرديّ.
- **الإجراء**: استخراج من التقرير الثالث + تكملة الناقص: عدد الـ Requests
  الكلّيّ (شبكة + API) لكل صفحة تمثيليّة عبر `preview_network` على بناء
  إنتاجيّ نظيف.
- **الوقت المتوقَّع**: 45 دقيقة – 1 ساعة.
- **المخاطرة**: صفر.
- **التأثير**: خطّ أساس رقميّ كامل وموحَّد.
- **الملفّات المتأثّرة**: ملفّ جديد `docs/roadmap/baselines/BASELINE-METRICS.md`
  يجمع: First Load JS لكل مسار، حجم JSON خام لكل endpoint رئيسيّ (homepage/
  category/article/writer/videos/reels/broadcasts)، عدد الطلبات لكل صفحة
  تمثيليّة.
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: `BASELINE-METRICS.md` موجود ويحتوي رقماً واحداً
  محدَّداً (لا نطاقاً) لكل مقياس، مع تاريخ القياس ونوع البناء (Dev/Prod).

### 0.4 حفظ Baseline كامل قبل إغلاق Sprint 0

- **النوع**: Observability/Process.
- **السبب**: تجميع 0.1–0.3 في نقطة مرجعيّة واحدة موسومة بوضوح، تُستخدَم
  حرفيّاً كمصدر المقارنة في `POST-SPRINT-1-VALIDATION.md` المستقبليّ
  (انظر الخاتمة).
- **الوقت المتوقَّع**: 15 دقيقة (تنظيم/مراجعة نهائيّة فقط).
- **المخاطرة**: صفر.
- **التأثير**: يجعل "قبل/بعد" قابلاً للإثبات، لا انطباعيّاً.
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: commit واحد بعنوان واضح (`docs: establish Sprint 0
  performance baseline`) يحتوي كل ملفّات 0.1–0.3.

# Sprint 1 — أعلى عائد (ROI)، أقلّ مخاطرة، أخطاء حيّة + SEO أوّلاً

**تغييرات الترتيب بعد مراجعة CTO**: `React.cache()` (كانت 1.5) **انتقلت
لِـ Sprint 2** (تحسين أداء بحت، ليست خللاً/UX/SEO/فشل إنتاجيّ). `Category
Metadata` (كانت 2.3) **انتقلت إلى هنا** كأولويّة أعلى من حذف المكوّنات
الميتة. أُضيف بند جديد: **API Contract Snapshot**.

### 1.1 التحقّق من فرضية تضخّم الترميز قبل أي تنفيذ (Unicode Encoding)

- **النوع**: Investigation (لا تنفيذ).
- **السبب**: التقرير الثالث قاس تضخّماً 1.78–2.1× في حجم استجابات JSON
  الخام. الأثر على *نقل الشبكة* يعتمد على ضغط الإنتاج، غير مؤكَّد بعد.
- **خطوات التحقّق**:
  1. تأكيد إعداد Cloudflare الفعليّ لضغط `/api/*` (Brotli/gzip).
  2. قياس مباشر: `curl -H "Accept-Encoding: br, gzip" -D - <production-domain>/api/v1/ar/homepage`، قراءة `Content-Encoding`
     والحجم الفعليّ بعد الضغط، مقارنةً بالخام.
  3. توثيق النتيجة وتعديل تقدير أثر 1.1-ب بناءً عليها.
- **الوقت المتوقَّع**: 30–60 دقيقة.
- **المخاطرة**: صفر — فحص فقط.
- **التأثير**: يُحدِّد أولويّة 1.1-ب الفعليّة (شبكة+معالجة أم معالجة فقط).
- **الملفّات المتأثّرة**: لا شيء (تحقّق فقط).
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: إجابة موثَّقة صريحة على السؤالين (هل Brotli/gzip
  مفعَّل على `/api/*`؟ ما الفرق الفعليّ بعد الضغط بالبايت؟).

### 1.2 تنفيذ `JSON_UNESCAPED_UNICODE` (بعد إتمام 1.1 فقط)

- **النوع**: Optimization (Backend).
- **السبب**: نتيجة 1.1.
- **الوقت المتوقَّع**: 15–30 دقيقة.
- **المخاطرة**: منخفضة جدّاً — تغيير ترميز متطابق دلاليّاً 100%.
- **التأثير**: تقليص كل استجابة JSON عربيّة تقريباً بنسبة ~45–55%
  (معالجة دائماً؛ شبكة حسب 1.1).
- **الملفّات المتأثّرة**: نقطة تصيير JSON المركزيّة في Laravel.
- Backend؟ **نعم** — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ نعم.
- **معايير القبول**: `curl` لأي endpoint عامّ يُظهِر نصّاً عربيّاً خاماً
  (UTF-8) لا تسلسلات `\uXXXX`؛ حجم استجابة الرئيسية والمقال منخفض بمقدار
  مطابق لنسبة 1.1's القياس (±5%)؛ صفر اختلاف في أي اختبار frontend قائم.

### 1.3 API Contract Snapshot (بند جديد — شبكة أمان قبل أي تعديل)

- **النوع**: Safety-net / Observability.
- **السبب**: مشروع Headless مثل AlphaCMS يعتمد كليّاً على ثبات عقد الـ
  API بين Laravel والواجهة. أي تعديل لاحق (Sprint 1 وما بعده) قد يُغيِّر
  شكل استجابة بصمت دون أن يُلاحَظ فوراً.
- **الإجراء**: حفظ استجابة JSON خام كاملة (قبل أي تعديل في هذه الخارطة)
  لكل من: `/{locale}/homepage`، `/{locale}/categories`،
  `/{locale}/articles/{slug}` (مقال حقيقيّ)، `/{locale}/writers/{id}`،
  `/{locale}/videos`، `/{locale}/videos/{slug}`، `/{locale}/reels`،
  `/{kind}` و`/{kind}/{slug}` للبثّ. كل استجابة تُحفَظ كملفّ `.json` باسم
  واضح + طابع زمنيّ، ويُفضَّل ربطها باختبار عقد آليّ (contract test) بسيط
  يقارن الحقول (لا القيم) لو توفّر وقت لاحقاً.
- **الوقت المتوقَّع**: 1–1.5 ساعة (7 endpoints).
- **المخاطرة**: صفر — قراءة/حفظ فقط.
- **التأثير**: **مرتفع طويل الأمد** — أي تغيير مستقبليّ في شكل العقد
  (إضافة/حذف/إعادة تسمية حقل) يظهر فوراً بمقارنة بسيطة، بدل اكتشافه
  لاحقاً عبر خلل صامت في الإنتاج (تماماً كما حدث مع `is_featured` أو
  `Video.media.renditions` المكتشَفَين في التقارير السابقة).
- **الملفّات المتأثّرة**: مجلَّد جديد
  `docs/roadmap/baselines/api-contracts/` (7 ملفّات JSON + فهرس).
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ لا.
- **معايير القبول**: 7 ملفّات JSON محفوظة، كل واحد يحمل تاريخ الحفظ
  ومعرِّف الـ commit الحاليّ في اسمه أو رأسه.

- Backend؟ محتمل — Frontend؟ **نعم** — Migration؟ محتمل — إعادة اختبار؟
  شامل.
- **معايير القبول**: لا ينطبق قبل اتّخاذ قرار المضيّ من الإدارة/المنتج.

### 3.9 اعتماد قياس Cold-vs-Warm Cache كممارسة قياسيّة دائمة

- **النوع**: Process / Methodology.
- **السبب**: أي رقم أداء مستقبليّ يجب تمييز الحالتين، لا افتراض أنّ رقماً
  واحداً يمثِّل "الوضع الحقيقيّ".
- **الوقت المتوقَّع**: لا وقت تنفيذ — سياسة تُطبَّق مستقبلاً على أي بند
  أداء جديد.
- Backend؟ لا — Frontend؟ لا — Migration؟ لا — إعادة اختبار؟ ينطبق على
  كل بند أداء مستقبليّ.
- **معايير القبول**: لا ينطبق (سياسة، لا تسليم منفرد).
