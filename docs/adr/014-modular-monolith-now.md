# ADR-014: مونوليث معياريّ (Laravel) الآن؛ استخراج خدمات عند الحاجة خلف الأحداث

## القرار
Laravel يبقى **مونوليثاً معياريًّا** واحدًا يخدم Content/Editorial/الكتابة
+ API + Projectors — منظَّماً داخليًّا بحدود Bounded Context واضحة (ADR-008)
لكن **منشوراً** كوحدة واحدة. استخراج أيّ سياق إلى خدمة (microservice)
منفصلة يحدث **فقط عند الحاجة الفعليّة**، خلف عقود الأحداث/API الموجودة
أصلاً.

## السبب
تجنّب تعقيد microservices التشغيليّ (شبكة، اكتشاف خدمات، معاملات موزَّعة،
مراقبة موزَّعة) في مرحلة لا يبرّره فيها حجم الفريق أو حِمل نظام محدَّد.
الحدود المعماريّة (Bounded Contexts + أحداث) مصمَّمة من اليوم بحيث
الاستخراج لاحقاً **لا يتطلّب إعادة تصميم**، فقط نقل الكود خلف نفس العقد.

## البدائل المرفوضة
- **microservices فوريّة** — تعقيد تشغيليّ مبكر (بنية تحتيّة، مراقبة،
  نشر منفصل لكلّ خدمة) بلا حاجة فعليّة تبرّره بعد.
- **مونوليث دائم** (بلا حدود داخليّة واضحة، بلا نيّة استخراج مستقبليّة) —
  يُنتج "big ball of mud" بمرور الوقت يصعب فكّه لاحقاً عند الحاجة الفعليّة
  للاستخراج.

## الأثر
استخراج أيّ سياق (مثلاً Search أو AI) إلى خدمة منفصلة لاحقاً يصبح عمليّة
منخفضة الألم — الحدود والعقود (أحداث + API) موجودة أصلاً؛ الاستخراج نقلٌ
للتنفيذ خلف نفس العقد، لا إعادة تصميم للتكامل.
