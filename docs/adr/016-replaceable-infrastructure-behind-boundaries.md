# ADR-016: بنية تحتيّة قابلة للاستبدال خلف حدود واضحة

## القرار
كلّ مكوّن بنية تحتيّة قابل للاستبدال (محرّك بحث، طابور/bus الرسائل، مخزن
كاش، مزوّد نموذج AI) يُستخدَم دائماً **خلف حدّ/عقد صريح** (مثل SearchAPI،
AIGateway، واجهة الـprojection) — لا رَبْط مباشر بمزوّد بعينه من كود
التطبيق.

## السبب
عمر المنصّة المخطَّط له (رؤية ٥ سنوات فأكثر) يعني أنّ أيّ مكوّن بنية
تحتيّة بعينه (Meilisearch، Redis، Claude كمزوّد AI) قد يحتاج استبدالاً
لاحقاً (نموّ الحجم، تغيّر تسعير المزوّد، ظهور بديل أفضل) — الحدّ الصريح
يجعل هذا الاستبدال محليًّا، لا كاسحاً لكامل التطبيق.

## البدائل المرفوضة
- **ربط مباشر بمزوّد بعينه** (كود التطبيق يستدعي Meilisearch/Claude/Redis
  مباشرة في كلّ مكان يحتاج القدرة) — يجعل استبدال أيّ مكوّن لاحقاً عمليّة
  إعادة كتابة واسعة عبر قاعدة الكود بأكملها.

## الأثر
ترقية أو استبدال أيّ مكوّن بنية تحتيّة (مثلاً Meilisearch → OpenSearch،
أو Redis → Dragonfly، أو مزوّد AI واحد → آخر) يحدث **خلف الحدّ** دون كسر
أيّ كود تطبيقيّ يستهلك القدرة عبر العقد الثابت.
